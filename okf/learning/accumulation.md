# 목적

OKF는 읽기 전용 참고 문서가 아니라 지속적으로 갱신되는 지식 번들이다. 반복 사용 가치가 있는 사실·규칙·결정·체크리스트는 작업 중 확인되는 시점에 OKF에 반영한다.

# 기록 대상

OKF에 기록한다:
- 여러 세션에서 재사용할 규칙·절차·체크리스트.
- 사용자가 명시적으로 확정한 영속 결정.
- 반복 오류의 근본 원인과 재발 방지 규칙.
- 도구 정책, 보안 정책, 코딩 스타일처럼 향후 행동을 바꾸는 지식.
- 게임·서버·SaaS 등 도메인별 판단 기준.

OKF에 기록하지 않는다:
- 현재 작업의 진행 상태·임시 TODO·일회성 디버깅 가설.
- API 키·토큰·PII·내부 접속정보.
- 특정 프로젝트에만 맞는 사실. 단, 전역 규칙으로 승격할 가치가 있으면 범위를 명시한다.
- 검증되지 않은 추측. 출처나 실제 코드로 확인한 뒤 기록한다.

# 기록 위치

1. 영속 학습의 소스 오브 트루스는 설정 레포의 소스 OKF다. 배포본 OKF만 직접 고치면 다음 setup 재배포 때 사라진다.
2. 개인별 누적 지식은 소스 OKF의 `/learned/` 아래에 둔다. 관리 번들 concept(예: `/workflow.md`, `/security/*`) 자체를 바꿔야 할 때는 소스 파일을 수정한다.
3. 같은 주제를 새 파일로 중복 생성하지 않는다. 기존 concept을 갱신할 수 있으면 갱신한다.
4. 새 concept을 만들면 같은 디렉터리의 `index.md`와 루트 `/index.md`에 절대 링크(`/path.md`)로 추가한다.
5. 변경 이력은 `/log.md`에 날짜별로 남긴다.
6. 소스 OKF를 수정한 뒤 setup을 재실행해 배포본으로 동기화한다.

# 절차

1. 학습 후보를 발견하면 소스 OKF의 `/learned/`와 기존 OKF에서 같은 주제의 concept을 찾는다.
2. 사실·결정·규칙 중 무엇인지 분류한다.
3. 보안·개인정보·프로젝트 한정성 여부를 점검한다.
4. 소스 OKF에 반영한다. 소스 레포 경로를 확인할 수 없으면 메모리에 임시로 남기고, 설정 정비 세션에서 OKF로 승격해야 한다고 사용자에게 알린다.
5. 관련 `index.md`와 `/log.md`를 갱신한다.
6. Google OKF v0.1 규칙을 확인한다: 비예약 concept은 YAML frontmatter와 non-empty `type` 필드가 있어야 하고, `index.md`·`log.md`는 예약 파일이다.

# 판단 기준

기록한다:
- 같은 지시를 다시 받지 않게 하는 정보.
- 같은 실수를 막는 정보.
- 사용자의 장기 선호·작업 방식에 영향을 주는 정보.
- 보안·데이터 손실·운영 리스크를 줄이는 정보.

기록하지 않는다:
- 한 번 쓰고 버릴 파일명·라인 번호·실험 결과.
- 검증이 끝나지 않은 추정.
- 외부 문서 링크만 있고 요약·판단이 없는 빈 항목.
