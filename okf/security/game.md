# 게임 클라/서버 보안

## 서버 권위 원칙
- 클라이언트 데이터는 전부 불신 — 서버에서 모든 게임 로직 검증.
- 클라이언트는 입력(커맨드)만 전송, 결과는 서버가 계산. 클라이언트가 보내는 좌표/수치/상태를 그대로 반영 금지.
- 검증 실패 입력은 권위 상태·last-known-good·통계·delta baseline·replay window를 갱신하지 않는다.
- 서버가 보정·정규화·재계산한 최종 accepted 값을 저장·시뮬레이션·브로드캐스트·송신자 보정·관전자 전파 전체에 사용한다.
- host/listen-server/local fast path도 원격 클라이언트와 같은 권한·검증·정규화 불변조건을 만족해야 한다.

## 보안과 반응성의 균형
서버 권위를 지키면서 클라이언트 반응성 확보:
- 클라이언트 예측(prediction): 커맨드 전송 즉시 선반영, 서버 응답으로 보정.
- 불일치 시 롤백/보정(reconciliation): 서버가 거부하면 서버 기준으로 되돌림.

| 분류 | 예시 | 전략 |
|------|------|------|
| 치명적(자원/재화 변동) | 구매·거래·유닛 생산 | 서버 확인 후 반영(낙관적 예측 금지) |
| 중요(게임 상태 변경) | 이동·전투·건설 | 클라이언트 예측 + 서버 보정 |
| 비치명적(표시/연출) | UI 애니메이션·카메라 | 클라이언트 단독 처리 |

- 서버 권위는 모든 행동에 유지하되, 동기 확인 대기는 치명적 행동에 한정 — 나머지는 예측+보정으로 반응성 확보.
- 안티패턴(금지): 모든 유저 행동을 서버 동기 검증(레이턴시 폭증), 불필요한 보안 처리로 프레임 드랍·로딩 히치 유발.

## 게임 커맨드 검증
- 커맨드 유효성: 이동 범위·자원 충분·쿨다운·턴 순서.
- 상태 전이 합법성: 현재 상태에서 해당 액션 가능 여부.
- 타이밍: 액션 rate limiting, 비정상 속도 탐지.
- 커맨드 중복/재전송: 구매·거래·보상·건설 등 비가역 명령은 command/transaction ID와 deduplication window로 멱등성을 보장한다.

## 경제 익스플로잇 방지
- 자원 생성/소비는 반드시 서버 트랜잭션으로. 레이스 컨디션 자원 복제 방지(동시 요청 직렬화). 정수 오버/언더플로우 검증.
- 최종 잔액·보상 상태와 이벤트 발행은 하나의 커밋 경계로 처리하고 부분 성공·중복 소비·중복 보상을 막는다.

## 세션/통신 보안
- 리플레이 공격 방지: session generation·sequence·command ID·허용 window를 검증하고 재접속 후 이전 generation 패킷을 자동 수락하지 않는다.
- 세션 재개 증명은 만료·회전·폐기가 가능해야 하며 로그에 남기지 않는다. 재개 시 identity·권한·replay window를 다시 결합하고 이전 연결과 동시 재개의 정책을 명시한다.
- 패킷 위변조 탐지와 전송 기밀성은 검증된 프로토콜을 사용한다. 비정상 연결·resume·resync 요청은 rate limit하고 자원 상한을 둔다.

## 정보 가시성·관전자
- interest management는 성능 최적화이고 fog of war·비공개 상태는 인가 경계다. 플레이어가 알 권한이 있는 정보만 서버에서 선별해 전송한다.
- 클라이언트에 비밀 상태를 전송한 뒤 UI·렌더러에서 숨기는 방식은 정보 보호가 아니다.
- 권위 보정·소유권 이전·observer 변경 후에도 모든 관전자는 허용된 동일 accepted 상태로 수렴해야 한다.

## 게임 클라이언트 보안
- 로컬 세이브·설정·패킷 등 클라 데이터를 무검증 신뢰 금지 → 서버 재검증.
- 재화·전투 결과·확률 등 민감 판정을 클라에서 확정하지 않는다(서버 권위). 위변조·메모리 변조 탐지·비정상 패턴 차단.
